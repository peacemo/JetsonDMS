## Module Designs

### Camera Module (MIPI CSI) - Fast Prototype

Goal:

- Detect and load a MIPI CSI camera
- Capture one frame
- Release camera resources safely

Design principle:

- Keep the first prototype simple and reliable
- Use OpenCV `VideoCapture` with a GStreamer pipeline string
- Avoid advanced threading or buffering in the first version

Input:

- Camera index (default `0`)
- Optional capture settings (width, height, fps)

Output:

- `cv::Mat` frame for success
- Error/status code for failure

Public API proposal (C++):

- `bool openCamera(int camIndex, int width, int height, int fps)`
- `bool captureFrame(cv::Mat& outFrame)`
- `void closeCamera()`

Internal workflow:

1. Detect and load camera
2. Validate camera state
3. Capture a frame
4. Release resources

#### 1) Detect and load MIPI CSI camera

Procedure:

- Build a GStreamer pipeline string for MIPI CSI camera input
- Try `cv::VideoCapture(pipeline, cv::CAP_GSTREAMER)`
- If it fails, return a camera-open failure status
- If it succeeds, mark module state as ready

Example pipeline for Jetson prototype:

`nvarguscamerasrc sensor-id=0 ! video/x-raw(memory:NVMM), width=1280, height=720, framerate=30/1 ! nvvidconv ! video/x-raw, format=BGRx ! videoconvert ! video/x-raw, format=BGR ! appsink`

Notes:

- This is the fastest practical path on Jetson for CSI camera bring-up
- Keep `sensor-id` configurable to support multiple cameras later

#### 2) Capture one frame

Procedure:

- Check if `VideoCapture` is opened
- Read one frame via `cap.read(frame)`
- Validate frame is not empty
- Return success/failure status

Minimal behavior for prototype:

- Single-shot capture only
- No retry loop in first version
- If capture fails, report error immediately

#### 3) Release camera resources

Procedure:

- Check if capture object is active
- Call `cap.release()`
- Clear internal ready flag

Expected result:

- Camera handle is closed cleanly
- Re-open in later calls is supported

#### Failure handling (prototype level)

- Open fail: camera not found, pipeline invalid, permission issue
- Capture fail: stream timeout, empty frame, temporary device issue
- Close fail: ignore hard failure but log warning

Recommended status enum:

- `CAMERA_OK`
- `CAMERA_NOT_FOUND`
- `CAMERA_OPEN_FAILED`
- `CAMERA_FRAME_EMPTY`
- `CAMERA_NOT_READY`

#### Prototype scope and non-goals

Included now:

- Basic MIPI CSI detection/open
- One-frame capture
- Resource release

Not included in first prototype:

- Auto-recovery and reconnect logic
- Multi-thread capture pipeline
- Exposure/white-balance controls
- Performance tuning and zero-copy optimization

This keeps implementation time short while making the camera module immediately usable by the vision pipeline.
